- Target remaining hard VM compiler opcode families with broad semantic cases rather than file-specific fixtures:
  - dynamic scope/ref ops: `with_make_ref`, `with_get_ref`, `with_get_ref_undef`, `make_*_ref`, `get_ref_value`, `put_ref_value` (plain `with_get_var`/`with_delete_var`, `with_put_var`, `apply_eval`, and uninvoked `import` bytecode are covered; `with_get_ref` probes currently crash, and naive `with_make_ref` lowering is not stack-compatible with `insert3`/`put_ref_value` bytecode)
  - remaining stack/control edges: `nip1`, generic `goto`, and non-trivial `gosub`/`ret`/finally paths (`dup1`, `nip`, `perm3/5`, `rot3l/3r/4l/5l`, and `swap2` are now covered through logical assignment, destructuring, iterator, and super cases)
  - remaining local/argument mutation forms: `get_loc`, `put_arg*`, `put_loc`, `set_loc*`; `add_loc`, `inc_loc`, and `dec_loc` are now covered; very-wide local probes hit BEAM arity limits, so avoid keeping them until lowering handles large frames differently
  - async/generator invocation semantics: uninvoked async/generator/import bytecode is covered for compile/opcode coverage, but invoked `await`, `yield`, `yield_star`, `return_async`, and dynamic `import()` still need semantic alignment before they can be correctness-audited
- Improve `vm_compiler_opcode_coverage` to report total opcode universe, percent coverage, and missing opcodes by family so coverage improvements are measurable.
- Fix semantic gaps before adding these cases to the invoked audited corpus:
  - `with` assignment needs branch-aware/property-reference lowering compatible with `insert3`/`put_ref_value`
  - derived constructors returning primitives should surface the same JS throw as the interpreter rather than crashing
  - invoked async/generator top-level results currently normalize differently between compiler and interpreter
  - invoked dynamic `import()` currently diverges in rejected promise error stack/message handling
  - computed class values expose `define_array_el`/`define_class_computed` lowering gaps
  - custom iterator loop coverage currently matches compiler/interpreter at `0`, which suggests a shared iterator runtime semantic gap outside the compiler-differential objective
- If corpus expansion exposes semantic bugs, prefer routing risky BEAM-specialized arithmetic/bitwise paths through JS runtime helpers over preserving unsafe raw BEAM operations.
